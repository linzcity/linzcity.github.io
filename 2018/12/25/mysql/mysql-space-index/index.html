直接存储经纬度信息，查找时带着经纬度，按照公式查找指定范围内的数据
想法最直接、实现也最容易，但是效率很低。

方案2：（数据库+java工具类处理）

区间查找
使用到com.spatial4j

com.spatial4j

具体思路：
表中使用两个字段存储经度和纬度，提前计算出经纬度的范围，然后在这两个字段上加上索引，提高搜索性能。
计算出经纬度的范围：已知条件是移动设备所在的经纬度，还有满足业务要求的半径，这很像初中的一道平面几何题：给定圆心坐标和半径，求该圆外切正方形四个顶点的坐标。而我们面对的是一个球体，公式较复杂，便可以使用spatial4j工具类来计算。

方案3：同样使用和2相同的工具类，不一样的原理：geohash

原理较复杂，大概就是：geohash算法把二维的经纬度编码成一维的字符串，它的特点是越相近的经纬度编码后越相似，所以可以通过前缀like的方式去匹配周围的商户
故数据库增加一个字段geo_code，来存储每个商户的geohash编码，并且建立索引
经纬度转为geohash编码
spatial4j提供了一个工具类GeohashUtils.encodeLatLon(lat, lon)，默认精度是12位，网上有geo_code长度与距离对照表。一般为5或6
sql语句：
SELECT id, name FROM customer WHERE geo_code LIKE CONCAT(?, ‘%’);
setString(1,geo_code)
研究一下geohash后容易发现，有一个很特殊但不得不面对的：边缘问题> geohash最终是在地图上铺上了一个网格，共9个，每一个网格代表一个geohash值，当传入的坐标接近当前网格的边界时，用上面的搜索方式就会丢失它附近的数据
解决方案：
我们查询时，除了使用当前位置的geohash编码进行匹配外，还使用周围8个网格的geohash编码，这样可以避免这个问题。
